# Wireframe LiveView with DaisyUI (light/dark)

## Goal

Build functional, wireframe-style screens quickly in LiveView

## House rules

- Use and enhance existing components (e.g. CoreComponents)
- Prefer **DaisyUI components** (buttons, cards, forms, tables) to keep markup simple.
- Use Tailwind utilities for layout/spacing
- Ensure UI works in light and dark themes
- Prioritize **functionality over aesthetics** (wireframe layouts, minimal custom styling).
- **Only `app.js` and `app.css` bundles** are supported; do not add vendor `<script>`/`<link>`.
- **Never** write inline `<script>...</script>` in HEEx. If JS needed, use **LiveView colocated hooks** (`<script :type={Phoenix.LiveView.ColocatedHook} ...>`).
- Page templates **must** start with `<Layouts.app flash={@flash} ...>` (and pass `current_scope` and `current_user`).
- Phoenix v1.8 note: `<.flash_group>` is defined in `Layouts` and **forbidden** outside `layouts.ex`
- `AppWeb.Layouts` is aliased in `app_web.ex`, so you can use `Layouts` in templates without another alias.
- Use `<.link navigate={...}>` / `<.link patch={...}>` (never `live_redirect` / `live_patch`).
- Use `<.form for={@form} id="...">` and `<.input field={@form[:field]} ...>` (never pass a changeset to the template).
- Ash domains should have `extensions: [AshPhoenix]` to enable form_to_*.
- Forms should use `form_to_*` code interface functions generated by `define`. Avoid direct `AshPhoenix.Form.for_create/for_update` in LiveViews.
- Prefer `<.icon ...>` from `core_components.ex` for icons (never call `Heroicons` modules directly).
- If you set `<.input class="...">`, you override defaults (no default classes are inherited) — your custom class must fully style the input.
- Give key elements **unique DOM IDs** if they only appear once per page (e.g. forms, containers, major buttons). Give non-unique elements descriptive data- tags to enable precise selection in tests.

## Troubleshooting

### Missing `current_scope` assign

- If you see errors about a missing `current_scope` assign, you likely:
  - Put routes in the wrong `live_session` (not following the authenticated routes patterns), and/or
  - Forgot to pass `current_scope` into `<Layouts.app ...>`
- Fix by moving the routes into the correct `live_session`, and passing `current_scope` to `<Layouts.app>` where required.

## Light/dark: project-specific guidance

This repo’s CSS config:

- DaisyUI ships with **two themes**: `"light"` (default) and `"dark"` (prefers dark).
- Dark mode is controlled via the **`data-theme="dark"`** attribute (there is a Tailwind `dark:` variant wired to `[data-theme=dark]`).

## JS and CSS guidelines

- Focus on rapid prototyping: functional wireframes over bespoke design
- Tailwind v4 uses `@import "tailwindcss" source(none);` and `@source ...` directives (no `tailwind.config.js` required)
  - Keep the `assets/css/app.css` import/source directives intact (don’t “simplify” them)
  - In this repo, `assets/css/app.css` includes sources like:
    - `@source "../css";`
    - `@source "../js";`
    - `@source "../../deps/ash_authentication_phoenix";`
- Never use `@apply` in raw CSS
- Only the `app.js` and `app.css` bundles are supported:
  - Do not add vendor `<script src>` / `<link href>` tags in layouts/templates
  - Import vendor dependencies into `app.js` / `app.css` instead
  - Never write inline `<script>...</script>` in templates (use LiveView colocated hooks)

Practical implications for wireframes:

- Prefer DaisyUI semantic tokens/classes (`bg-base-100`, `text-base-content`, `border-base-300`, `card`, `btn`, `alert`, etc.) so colors adapt automatically.
- If you add Tailwind color utilities, prefer token-ish ones (like `text-base-content/70`) over hard-coded grays to avoid theme mismatch.
- If a page needs an explicit theme toggle later, implement it by setting `data-theme` on a parent/root element (don’t invent a new CSS approach).

## Workflow: build a new screen fast

### 0) Reuse and evolve `core_components.ex`

Before writing custom markup:

- Start with `core_components.ex` and use the existing function components (`<.input>`, `<.button>`, `<.table>`, `<.icon>`, etc.).
- It’s OK to **update existing core components** to better match the desired wireframe style (e.g., adjust daisy/tw classes) when it improves consistency across the app.
- If you need a reusable UI primitive that doesn’t exist yet, **create a new function component** in `core_components.ex` rather than duplicating markup across views.
- **Rule of 2**: once the same markup pattern appears in **two** places, extract it into a shared function component (typically in `core_components.ex` unless it’s feature-specific).

### 1) Pick the UI skeleton (DaisyUI-first)

Use these building blocks:

- **Page container**: `container mx-auto max-w-5xl px-4 py-6`
- **Header row**: title + actions aligned with `flex items-center justify-between gap-4`
- **Main card**: `card bg-base-100 border border-base-300`
- **Card body**: `card-body` with `gap-4`
- **Primary actions**: `<.button variant="primary">...</.button>` (or `class={["btn", "btn-primary"]}`)

### 2) Add layout and spacing with Tailwind

Use Tailwind utilities for:

- **Layout**: `flex`, `grid`, `gap-*`, `items-*`, `justify-*`
- **Sizing**: `w-full`, `max-w-*`
- **Spacing**: `p-*`, `py-*`, `px-*`, `mt-*`

Avoid:

- Custom CSS and `@apply`
- Over-design (shadows, gradients, complex animations)

### 3) Forms: use the standard pattern

- In the LiveView, build forms via domain/resource code-interface `form_to_*` helpers, then convert with `to_form/1`.
- Prefer this pattern:
  - `MyDomain.form_to_create_thing(actor: current_user, as: "thing") |> to_form()`
  - `MyDomain.form_to_update_thing(record, actor: current_user, as: "thing") |> to_form()`
- Do not call `AshPhoenix.Form.for_create/for_update` directly when a `form_to_*` helper exists.
- In the template:
  - `<.form for={@form} id="my-form" phx-change="validate" phx-submit="save">`
  - `<.input field={@form[:name]} label="Name" />`

If you override `<.input class="...">`, you must fully style it (defaults won’t apply).

### 4) Lists/collections: streams when it matters

If rendering a collection in LiveView, strongly consider **LiveView streams**:

- Parent element must have `phx-update="stream"` and a stable DOM id.
- Items must use `@streams.name` and `id={id}`.
- Don’t `Enum.filter` a stream; refetch and `stream(..., reset: true)` instead.

### 5) JS needs: colocated hooks only

If the UI needs lightweight JS:

- Use colocated hooks: `<script :type={Phoenix.LiveView.ColocatedHook} name=".MyHook"> ... </script>`
- Hook names must start with a dot (e.g. `.PhoneNumber`).
- If a hook manages its own DOM, add `phx-update="ignore"` and a unique `id`.

### 6) Keep structure healthy as screens grow

Use this escalation path to avoid large, brittle LiveViews:

1. **Start simple**: one LiveView + function components for display-only blocks.
2. **Promote stateful workflows** (modals/forms with validate+submit lifecycle) into `Phoenix.LiveComponent`s.
3. **Keep the parent LiveView as orchestrator**:
   - routing/params and tab/filter state
   - cross-component coordination
   - real-time subscriptions (`handle_info/2`)
   - shared table/list refresh orchestration
4. **Keep each LiveComponent local**:
   - form initialization in `update/2`
   - `handle_event("validate", ...)` and `handle_event("submit", ...)`
   - local UI state (open/close/error display)

#### Parent-child contract (required)

- Prefer a clear **assigns-in / messages-out** contract.
- Parent passes only required assigns (`current_user`, record id, visible state, options).
- Child notifies parent with explicit tuples, e.g.:
  - `send(self(), {:ui_updated, %{source: :create_item}})`
  - `send(self(), {:close_modal, :create_item})`
- Parent decides refresh scope and flash policy.

#### Update strategy (important)

- For **local submits** in the current session: refresh from parent immediately after success.
- Keep **PubSub** for cross-session/domain-driven updates.
- Do not use PubSub as an intra-page event bus between parent and child components.

#### URL state and routability (important)

- Prefer the URL as the source of truth for durable UI state so refresh/back/forward preserve context.
- Keep state like active tab, filters, selected record, and open detail/edit context in params/path (not only assigns).
- In LiveViews, pair `handle_params/3` with `push_patch` and `<.link patch={...}>` for state transitions.
- Critical view state should be reconstructable from URL alone after page reload.

For CRUD-style UIs, prefer explicit, nested routes over opaque query-only flows:

- `/resources?tab=list`
- `/resources/new`
- `/resources/:resource_id`
- `/resources/:resource_id/edit`
- `/resources/:resource_id/subresources?order_by=name`
- `/resources/:resource_id/subresources/new`
- `/resources/:resource_id/subresources/:subresource_id`
- `/resources/:resource_id/subresources/:subresource_id/edit`

Use query params for orthogonal UI state (tab/sort/filter), and path segments for resource identity and CRUD mode.

#### Event naming and extraction triggers

- Namespace component events to avoid collisions (`"create:validate"`, `"create:submit"`).
- Extract before complexity spikes. Strong signals:
  - LiveView grows beyond roughly 400-600 lines
  - 3+ independent modal/form workflows
  - repeated open/close/validate/submit patterns
  - render function mixes orchestration and large UI blocks

#### Refactor safely

- Move one vertical slice first (one workflow end-to-end), validate, then repeat.
- Preserve stable DOM ids and `data-test` selectors during extraction to keep tests robust.

## Templates

### Page scaffold (HEEx)

Use as a starting point in a LiveView template:

```heex
<Layouts.app flash={@flash} current_scope={@current_scope} current_user={@current_user}>
  <div id="page-RESOURCE" class="container mx-auto max-w-5xl px-4 py-6">
    <div class="flex items-center justify-between gap-4">
      <div>
        <h1 class="text-xl font-semibold">TITLE</h1>
        <p class="text-sm text-base-content/70">Optional subtitle</p>
      </div>
      <div class="flex items-center gap-2">
        <.button variant="primary">Primary</.button>
        <.button class={["btn", "btn-ghost"]}>Secondary</.button>
      </div>
    </div>

    <div class="mt-6 card bg-base-100 border border-base-300">
      <div class="card-body gap-4">
        <!-- content -->
      </div>
    </div>
  </div>
</Layouts.app>
```

### Form-in-a-card (HEEx)

```heex
<div class="card bg-base-100 border border-base-300">
  <div class="card-body">
    <.form for={@form} id="resource-form" class="space-y-3" phx-submit="save">
      <.input field={@form[:name]} label="Name" />
      <.input field={@form[:description]} type="textarea" label="Description" />

      <div class="flex items-center gap-2 pt-2">
        <.button variant="primary" phx-disable-with="Saving...">Save</.button>
        <.link navigate={~p"/somewhere"} class="btn btn-ghost">Cancel</.link>
      </div>
    </.form>
  </div>
</div>
```

## When to consult docs

If you need a component pattern, look up DaisyUI components first (cards, navbar, tabs, alerts, drawer).
Use Tailwind docs only for layout/spacing/grid/flex when DaisyUI doesn’t cover it.
